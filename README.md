# cubalizator_bot

Telegram-фактчекер на GigaChat. Отвечает при упоминании `@cubalizator_bot` в любом чате, куда добавлен; при упоминании в ответе на чужое сообщение передаёт цитату в GigaChat.

## Локальная проверка

```bash
python -m venv .venv
# Windows: .venv\Scripts\activate
# macOS/Linux: source .venv/bin/activate
pip install -r requirements-dev.txt
pytest -q
```

Для локального webhook нужен публичный HTTPS URL. Основной вариант развёртывания — Yandex Cloud Serverless Containers.

## Переменные окружения

Скопируйте `.env.example` в `.env` только для локальной работы. `.env` не коммитить.

`ALLOW_REPLY_TO_BOT=false` означает: бот отвечает только при `@упоминании`. Если поставить `true`, он также продолжит разговор при обычном ответе на сообщение самого бота.

## Сборка Docker

```bash
docker build -t cubalizator:latest .
```

## Webhook

После публикации контейнера задайте локально переменные `TELEGRAM_BOT_TOKEN`, `PUBLIC_URL`, `WEBHOOK_PATH_SECRET`, `TELEGRAM_WEBHOOK_SECRET`, затем выполните:

```bash
python scripts/set_webhook.py
```

URL будет таким:

```text
https://<public-container-url>/telegram/webhook/<WEBHOOK_PATH_SECRET>
```

## Поиск по истории группы

Бот умеет отвечать на вопросы про переписку («на какую сумму Бен получил премию»,
«с кем Илюха тусил в Питере», «что Даня говорил про BMW»).

Как это работает:
1. При старте экземпляра функции один раз лениво читается `data/chat_history.json`
   и строится инвертированный индекс в памяти (BM25). Весь JSON в GigaChat не шлётся.
2. Из вопроса определяется человек (с учётом алиасов из `app/aliases.py`) и период
   («в этом году», «за лето», «в январе»). По ним отсекаются кандидаты.
3. По оставшимся сообщениям считается BM25; для денежных вопросов сообщения с
   суммами получают приоритет.
4. Топ-фрагменты (автор, дата, текст) передаются GigaChat как единственный источник.
   Если ничего не найдено — бот отвечает «В переписке этого не нашёл», не выдумывая.

Путь к файлу задаётся переменной `CHAT_HISTORY_PATH` (по умолчанию
`data/chat_history.json`). Если файл отсутствует или повреждён — приложение не
падает, обычные вопросы продолжают работать, а на вопрос по истории бот честно
скажет, что переписку сейчас не поднять.

### Обновить историю из свежего экспорта Telegram

Выгрузи чат через Telegram Desktop в JSON (`result.json`) и собери слим-версию:

```bash
python scripts/build_history.py result.json data/chat_history.json
```

Скрипт выкидывает фото, реакции и метаданные, оставляя автора, дату и текст —
файл становится примерно вдвое легче и быстрее грузится.

## Архитектура ответов по истории (двухпроходный поиск + анализ через GigaChat)

Главный принцип: локальный код только ищет и готовит контекст, все смысловые
выводы делает GigaChat.

Поток (см. `app/analysis.py`):
1. **Планирование запроса (GigaChat).** `analysis.plan_search()` шлёт вопрос в
   GigaChat с системным промптом `PLANNER_SYSTEM` и получает JSON-план: люди,
   прозвища, места, события, варианты написания, доп. формулировки, нужны ли
   reply-цепочки и соседи, и относится ли вопрос к истории. Если модель недоступна
   или вернула мусор — есть эвристический фолбэк, чтобы бот не падал.
2. **Сбор эпизодов (локально, без GigaChat).** `analysis._retrieve()` по плану
   гоняет `history.search_indices()` для вопроса и каждого термина, затем
   `retrieval.gather_episodes()` добавляет к каждому попаданию 7 сообщений до/после,
   сообщение-родитель (reply_to) и все ответы на него, склеивает пересекающиеся
   окна в эпизоды и убирает дубли. Параллельно `memory.search_memory()` достаёт
   подсказки из структурированной памяти.
3. **Смысловой анализ (GigaChat).** `analysis.analyze_once()` шлёт эпизоды и
   подсказки в GigaChat с промптом `ANALYZER_SYSTEM`. Модель определяет прозвища
   по reply-контексту, связывает сообщения в события, помечает уверенность
   (подтверждено / вероятно / недостаточно) и либо отвечает, либо просит второй поиск.
4. **Второй проход.** Если модель вернула `need_more`, оркестратор берёт её
   follow-up запрос, до-собирает эпизоды, объединяет их с прежними (дедуп по
   message_id) и снова зовёт анализ. Проходов не больше `max_passes` (по умолчанию 2).

Что осталось обычным поисковым кодом (никакого «понимания», только данные):
`app/history.py` (индекс, BM25, фильтры по автору/дате/суммам),
`app/retrieval.py` (соседи, reply-цепочки, эпизоды, дедуп),
`app/memory.py` (загрузка и поиск по `data/chat_memory.json`).

### Пример потока

Вопрос «Кого называют Корнишон?»: планировщик помечает прозвище и просит
reply-цепочки → локально находим сообщения со словом «корнишон» и вокруг каждого
поднимаем родителя и ответы (видно, к кому обращались) → GigaChat по этим цепочкам
делает вывод, кого так зовут, с пометкой уверенности.

Вопрос «Что ты знаешь про Хриплую и Питер?»: планировщик даёт варианты написания
(Хриплая/Хриплой) и место (Питер/СПб) → собираем эпизоды по обоим → если в первом
проходе связь неявная, GigaChat просит второй поиск (например, по Илюхе и дате из
найденного) → эпизоды объединяются, и только потом формируется ответ.

### Долговременная память

`data/chat_memory.json` строится один раз из всей переписки пакетами через GigaChat
(`scripts/build_memory.py`), а не регулярками. Схема: `people`, `aliases`, `events`,
`inside_jokes`, `relationships`; у записей — описание, участники, даты, `evidence`
(message_id) и `confidence`. В репозитории лежит seed с ростером и известными
алиасами; события/шутки/связи наполняются билдером:

```bash
python scripts/build_memory.py            # вся история (долго, много запросов)
python scripts/build_memory.py --limit 3000 --batch 150   # частично
```

Путь переопределяется через `CHAT_MEMORY_PATH`. Если файла нет или он битый —
бот работает без памяти, опираясь на исходные сообщения.
